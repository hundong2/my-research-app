# Issue #8 - IBM Spyre 가속기 PyTorch 통합 요약 및 분석

- Source: https://research.ibm.com/blog/pytorch-support-ibm-spyre
- Published: 2026-03-06
- Authors: Viji Srinivasan, Raghu Ganti, Mudhakar Srivatsa, Elpida Tzortzatos, Tabari Alexander, Vaidyanathan Srinivasan
- Focus: IBM Spyre 가속기를 PyTorch 생태계에 네이티브 통합하기 위한 2026년 상반기 로드맵 및 기술 전략

---

## 1) 한 줄 요약

IBM은 데이터플로우 가속기 Spyre를 **torch.inductor 확장 + 전용 컴파일러 계층(KTIR) + vLLM 플러그인** 형태로 PyTorch 생태계에 네이티브 통합하며, 개발 과정에서 파생된 프레임워크와 테스트 인프라를 오픈소스로 환원해 타 가속기 팀에도 혜택이 돌아가는 생태계 중심 전략을 추구한다.

---

## 2) IBM Spyre 하드웨어 특성

| 항목 | 내용 |
|---|---|
| AI 코어 수 | 32개 (링 연결 구조) |
| 연산 방식 | 혼합 정밀도 SIMD-Systolic 배열 |
| 실행 모델 | 프로그래머블 데이터플로우(Programmable Dataflow) |
| 메모리 모델 | 명시적 온칩 스크래치패드(Explicitly Managed Scratchpad) |

- **링 연결 코어 구조**: 코어 간 통신이 환형 토폴로지로 이루어져, 데이터 이동 패턴이 GPU의 SIMT 모델과 근본적으로 다름
- **스크래치패드 메모리**: GPU의 HBM/캐시 계층 구조 대신 소프트웨어가 직접 데이터 배치를 관리 → 컴파일러가 메모리 스케줄링까지 담당해야 함
- **데이터플로우 방식**: 연산 그래프 전체를 컴파일 타임에 타일 단위로 분해해 코어에 배분하는 정적 스케줄링이 핵심

---

## 3) 핵심 통합 전략: torch.inductor 확장

PyTorch의 기본 컴파일러 백엔드인 `torch.inductor`를 세 가지 방향으로 확장한다.

### 3-1) 타일 기반 텐서 레이아웃 (Tile-Based Tensor Layouts)
- 컴파일러가 블록 구조 데이터 이동을 코어 단위로 추론할 수 있도록 레이아웃 표현 추가
- 기존 스트라이드(Stride) 기반 텐서 표현으로는 링 연결 구조의 타일 분산 패턴을 표현하기 어렵기 때문

### 3-2) 멀티코어 작업 분할 패스 (Multi-Core Work-Division Passes)
- 런타임이 아닌 **컴파일 타임에** 32개 코어로의 타일 분배를 결정
- 정적 스케줄링을 통해 런타임 오버헤드를 최소화하고 예측 가능한 성능 확보

### 3-3) 스크래치패드 최적화 (Scratchpad Optimization)
- 명시적 온칩 메모리를 고려한 데이터 스케줄링 로직 추가
- 컴파일러가 어떤 데이터를 언제 스크래치패드에 올리고 내릴지 결정

**적용 대상 모델 범위:** Llama 3.1 8B ~ Granite 4 Hybrid 30B (2026 상반기 배포 목표)

## 4) 컴파일러 스택 구조

2단계 컴파일러 아키텍처를 채택해 PyTorch 통합 레이어와 하드웨어 최적화 레이어를 분리한다.

```
PyTorch 모델 (FX Graph)
        │
   torch.inductor
        │
  ┌─────────────────────────┐
  │  Stage 1: SuperDSC      │  ← 연산 하강(Lowering) 및 코드 생성 단일 진입점
  │  (Backend Compiler IR)  │    PyTorch 통합과 HW 최적화 경계 분리
  └──────────┬──────────────┘
             │
  ┌──────────▼──────────────┐
  │  Stage 2: KTIR           │  ← 커뮤니티 공개 스펙 (2026 상반기 발표 예정)
  │  (KernelTile IR)         │    타일 수준 표현으로 스케줄링 및 코드 생성
  └─────────────────────────┘
        │
   Spyre 하드웨어
```

- **SuperDSC**: IBM 내부 백엔드 IR. 연산 하강과 코드 생성의 단일 진입점 역할. PyTorch 통합 코드와 HW 최적화 코드의 결합을 방지
- **KTIR (KernelTile IR)**: 타 데이터플로우 가속기도 활용 가능한 범용 타일 수준 IR. 오픈 스펙 공개 계획으로 커뮤니티 표준화 목표

---

## 5) 런타임 및 분산 추론

- Spyre는 **아웃오브트리(Out-of-Tree) 확장**으로 등록: 디바이스 라이프사이클, 메모리 관리, 데이터 전송, 디스패치 모두 업스트림 코드 수정 없이 처리
- 목표 오버헤드: **직접 디바이스 접근 대비 5% 미만**
- 집합 통신(all-reduce, all-gather)은 `torch.inductor`를 통해 컴파일
- 향후 `torch.distributed` 및 `torch.comms`로 마이그레이션 예정

---

## 6) 프로덕션 추론: vLLM 통합

| 구성 요소 | 내용 |
|---|---|
| 통합 방식 | vLLM 플랫폼 플러그인 (업스트림 모델 구현 그대로 사용, 커스텀 포크 없음) |
| Spyre Attention Backend | 동일 시퀀스 길이 제약 제거 → 토큰 간 지연 감소 |
| Compile 캐싱 | `torch.compile` 아티팩트 캐싱 개선 → 시작 시간 수 초 수준으로 단축 |

- 업스트림 모델 구현을 그대로 사용해 커스텀 포크 유지 비용을 제거
- Spyre Attention Backend가 기존의 동일 시퀀스 길이 제약을 해소하여 실제 추론 시 inter-token latency 개선
- 컴파일 캐싱 개선으로 콜드 스타트 문제 해소

## 7) 테스트 인프라 (레이어드 테스트 피라미드)

```
[End-to-End vLLM 추론]
      ↑
[Top-Level 모델 품질 및 성능]
      ↑
[모듈 수준 테스트 (Attention, Normalization, Activation)]
      ↑
[Inductor 컴파일 및 하강 검증]
      ↑
[연산(Operation) 단위 정확도]
```

- **목표**: 야간 테스트 통과율 >95%, 전체 파이프라인 완료 시간 <3시간

## 8) 생태계 기여 계획 (Ecosystem Contributions)

IBM은 Spyre 통합 과정에서 개발된 3가지 결과물을 PyTorch 커뮤니티에 환원할 예정이다.

| 기여 항목 | 내용 | 의의 |
|---|---|---|
| **OpenReg Primitives** | 아웃오브트리 디바이스를 1급 시민으로 테스트 가능하게 하는 기반 요소 | 타 가속기 팀의 디바이스 등록 및 테스트 진입 장벽 낮춤 |
| **KTIR 일반화** | 데이터플로우 가속기 공통 타일 수준 IR 표준 | 다양한 비-GPU 가속기 간 컴파일러 레이어 공유 가능 |
| **아웃오브트리 CI 문서화** | 외부 가속기 팀을 위한 CI 구성 패턴 및 가이드 | 새 가속기 개발팀의 통합 시간 단축 |

## 9) 기술적 분석

### 9-1) GPU 중심 PyTorch 생태계에서 비-GPU 가속기의 통합 방식 전환
- 기존에는 비-GPU 가속기가 PyTorch에 통합되려면 업스트림 코드를 직접 수정하거나 독자적인 포크를 유지해야 했음
- IBM의 접근은 `OpenReg` + 아웃오브트리 확장을 통해 **업스트림 수정 없이 네이티브 수준의 통합**을 달성하는 패턴을 제시

### 9-2) 컴파일러 레이어 분리 전략의 타당성
- SuperDSC와 KTIR의 2단계 분리는 PyTorch 버전 변경 시 SuperDSC만 수정하면 되는 구조로 유지보수 비용을 낮춤
- KTIR을 별도 스펙으로 공개하면 Intel Gaudi, Graphcore IPU 등 타 데이터플로우 계열 가속기도 동일 IR을 재사용할 수 있는 생태계 기반 마련 가능

### 9-3) 스크래치패드 메모리 관리의 컴파일러 책임 이전
- GPU는 캐시 계층이 자동으로 데이터 이동을 처리하지만, Spyre는 소프트웨어가 명시적으로 관리해야 함
- 이를 컴파일러 수준에서 자동화하면 개발자가 하드웨어 세부 사항을 몰라도 최적 성능을 얻을 수 있어 채택 장벽이 낮아짐

### 9-4) vLLM 통합의 실용적 의의
- LLM 추론 서빙 사실상 표준인 vLLM과의 통합은 Spyre의 실제 배포 가능성을 확보하는 핵심 요소
- 커스텀 포크 없이 업스트림 모델을 그대로 사용한다는 점은 모델 업데이트 주기에 지장 없이 최신 모델을 Spyre에서 서빙할 수 있음을 의미

## 10) 한계와 주의사항

1. **스크래치패드 관리 복잡도**: 컴파일러가 자동화하더라도 메모리 용량 제약이 있는 모델에서는 타일 크기와 배치 크기 선택이 성능에 크게 영향을 미칠 수 있음
2. **정적 스케줄링의 유연성 제약**: 컴파일 타임에 작업 분배를 결정하므로, 동적 형상(Dynamic Shape)이 많은 워크로드에서는 재컴파일 비용이 발생할 수 있음
3. **KTIR 표준화 불확실성**: 커뮤니티 표준으로 채택되려면 타 가속기 업체의 참여가 필요하며, 2026 상반기 스펙 공개 이후 실제 채택 여부는 미지수
4. **5% 오버헤드 목표**: 직접 디바이스 접근 대비 5% 이하는 야심찬 목표로, 실제 워크로드 다양성에 따라 편차가 있을 수 있음
5. **블로그 기준 수치**: 공개된 성능 지표는 특정 실험 조건 기반이므로 범용적으로 재현된다고 가정하면 안 됨

---

## 11) 결론

IBM Spyre의 PyTorch 통합 전략은 단순한 하드웨어 지원 추가가 아닌, **비-GPU 가속기가 주류 AI 소프트웨어 스택에 진입하는 방법론적 청사진**을 제시한다. 핵심 가치는 두 가지다: ① torch.inductor 확장과 아웃오브트리 디바이스 등록을 통해 업스트림 오염 없이 네이티브 통합을 달성하고, ② KTIR과 OpenReg 등 범용 컴포넌트를 커뮤니티에 환원해 다음 가속기 통합 팀의 진입 비용을 낮추는 선순환 구조를 만드는 것이다. 성공 여부는 2026 상반기 KTIR 스펙 공개와 커뮤니티 채택 반응에 달려 있다.
