# Issue #4 - C2PA 2.3 Introduction 요약 및 상세 분석

- Issue: https://github.com/hundong2/my-research-app/issues/4
- Source: https://spec.c2pa.org/specifications/specifications/2.3/specs/C2PA_Specification.html#_introduction
- Scope: C2PA Specification 2.3의 Introduction(1.1~1.3) 중심 해석 + 핵심 용어(2.3 일부) 연계 분석

## 1) 한 줄 요약
C2PA는 콘텐츠의 "진위(truth)"를 판정하는 표준이 아니라, 누가/무엇을/언제/어떻게 만들고 수정했는지에 대한 출처(provenance)와 변환 이력을 서명 가능한 형태로 전달해, 소비자가 신뢰 판단을 할 수 있게 만드는 상호운용 표준이다.

## 2) Introduction 핵심 요약
1. **문제 정의**: 생성형 AI와 디지털 편집 확산으로 콘텐츠 조작 탐지가 어려워졌고, "출처 정보"를 일관되게 전달할 공통 규격이 필요해졌다.
2. **C2PA의 목적**: 콘텐츠 자체의 진실성 보증이 아니라, 제작/수정 이력과 주체 정보를 검증 가능한 방식으로 묶어 전달하는 것.
3. **신뢰 모델**: 게시자(publisher), 제작자(creator), 검증자(verifier), 소비자(consumer) 간에 공통 해석 가능한 메타데이터 체계를 제공.
4. **구현 형태**: 콘텐츠에 직접 내장(embedded)하거나 외부(원격) 참조 방식으로 연결된 매니페스트(manifest)와 서명(signature) 체계를 사용.
5. **상호운용성 목표**: 다양한 포맷(이미지/비디오/오디오 등), 다양한 도구(카메라, 편집기, 플랫폼) 간에 통합 가능한 방식 지향.

## 3) 구조적 해석 (왜 이렇게 설계됐는가)
1. **콘텐츠 진실 판정과 출처 증명의 분리**
- C2PA는 "이 콘텐츠가 사실인가"를 직접 판단하지 않는다.
- 대신 "어떤 행위가 어떤 순서로 있었는가"를 서명 체인으로 고정한다.
- 이 분리는 기술적/법적 리스크를 줄이고 표준 채택 가능성을 높인다.

2. **Claim + Assertion + Signature의 계층화**
- `Assertion`: 캡처 장치, 편집 액션, 생성형 AI 사용 여부 같은 개별 사실 조각.
- `Claim`: 어떤 assertion들이 한 번의 저작 행위를 구성하는지 묶는 단위.
- `Signature`: claim 무결성과 발행 주체를 암호학적으로 검증하는 장치.
- 계층화 덕분에 부분적 확장(새 assertion 유형 추가)과 버전 진화가 용이하다.

3. **내장형/원격형 매니페스트 병행 지원**
- 플랫폼별 파일 제약(재인코딩, 리사이징, 썸네일 변환 등)을 고려한 설계.
- 내장형은 휴대성, 원격형은 유통 유연성을 강화한다.
- 실제 서비스에서는 두 방식을 혼합해 가용성과 추적성을 동시에 확보할 수 있다.

4. **"신뢰 강제"가 아닌 "신뢰 신호 제공" 모델**
- C2PA는 결과를 강제하지 않고, 검증 가능한 신호를 제공한다.
- 최종 판단은 UI 정책/플랫폼 정책/사용자 맥락이 담당한다.
- 따라서 제품 품질은 "표준 준수" 자체보다 "검증 결과를 어떻게 설명하는 UX"에 크게 좌우된다.

## 4) 기술적으로 중요한 포인트
1. **무결성 경계(Integrity Boundary)**
- 어떤 바이트 범위를 해시/서명 대상으로 보는지가 핵심이다.
- 포맷 변환 파이프라인에서 이 경계가 흔들리면 검증 실패가 빈번해진다.

2. **액터 신원(Identity)과 인증서 체인**
- 서명 자체보다 인증서 신뢰 사슬(PKI) 관리가 운영 난이도를 결정한다.
- 만료/폐기/교체 정책을 설계하지 않으면 장기 검증 신뢰도가 급격히 떨어진다.

3. **데이터 최소화와 프라이버시**
- provenance 강화와 개인정보 최소화는 충돌할 수 있다.
- assertion 설계 시 공개 가능한 정보와 비공개 정보를 분리해야 한다.

4. **부분 손실 시나리오 대응**
- SNS 재압축, 스크린샷, 포맷 변환으로 매니페스트가 유실될 수 있다.
- "정보 없음"과 "변조됨"을 UX에서 구분해야 오탐 인식을 줄일 수 있다.

## 5) 실무 적용 체크리스트
1. 생성/편집 파이프라인의 각 단계에서 어떤 assertion을 생성할지 먼저 정의한다.
2. 서명키 수명주기(발급/회전/폐기/감사 로그)를 운영 정책으로 문서화한다.
3. 내장형 + 원격형 매니페스트를 서비스 특성에 맞춰 혼합 적용한다.
4. 검증 실패 원인(서명 불일치, 체인 불신뢰, 메타데이터 유실)을 사용자 메시지로 구분한다.
5. "C2PA 존재"를 진실성 보증으로 오해하지 않도록 UI 카피를 분리한다.

## 6) 한계와 주의점
1. C2PA는 "합성/조작 탐지 모델"의 대체제가 아니다.
2. 초기 원본에서 provenance가 누락되면 사후 복원은 제한적이다.
3. 플랫폼 간 재가공이 심할수록 메타데이터 유지율 문제가 커진다.
4. 제도/정책(표시 의무, 법적 효력)과 결합되지 않으면 사용자 신뢰 효과가 제한될 수 있다.
