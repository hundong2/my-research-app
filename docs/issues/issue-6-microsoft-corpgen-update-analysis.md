# Issue #6 - Microsoft CorpGen 업데이트 요약 및 분석

- Source: https://www.microsoft.com/en-us/research/blog/corpgen-advances-ai-agents-for-real-work/?hl=ko-KR
- Related publication: https://www.microsoft.com/en-us/research/publication/corpgen-learning-from-task-correction-feedback-for-language-model-based-code-generation/
- Focus: 실제 개발 워크플로우에서 사용할 수 있는 코드 에이전트용 데이터 생성/학습 방법(CorpGen)과 실험 결과 해석

## 1) 한 줄 요약
Microsoft의 CorpGen은 **실제 업무형 코딩 작업 데이터셋을 자동 생성하고**, 그 위에 학습한 모델이 기존 벤치마크 중심 학습 대비 더 긴 과제와 실제 버그 수정 맥락에서 강한 성능을 보인다는 점을 보여준다.

## 2) 핵심 내용 정리
1. **문제의식**: 기존 코딩 벤치마크는 짧고 독립적인 문제 비중이 커서, 실제 업무의 장기/다단계 작업을 충분히 반영하지 못한다.
2. **CorpGen 접근**: 인간 작성 태스크를 시작점으로 삼아 LLM이 데이터셋을 자동 확장하며, 단순 정답 생성이 아니라 "수정-피드백-개선" 흐름을 반영한다.
3. **데이터 특성**: 파일 수정, 테스트 실패 후 고치기, 요구사항 변경 대응 등 실제 엔지니어링 루프와 유사한 패턴을 포함한다.
4. **학습 효과**: 장문 컨텍스트에서 일관성 있게 코드를 고치는 능력과, 도구 기반 검증(test/lint 등) 친화성이 개선된다.

## 3) 블로그에 나온 주요 수치(핵심 지표)
1. **SWE-bench Lite 기준 에러율**: `16.7 -> 8.7`(상대적으로 약 48% 감소)
2. **코드 수정량(불필요 변경 억제)**: `15.2 -> 4.3`(약 3.5배 개선)
3. **평가 방식 이슈**: 기존 pass/fail 평가는 실제로는 유용한 부분 정답까지 실패로 처리하는 문제가 있으며, 사례 기준으로 사람 평가는 약 90%를 유용하다고 본 반면 자동 평가는 약 40%만 성공으로 집계

## 4) 기술적 분석
1. **"정답 생성"에서 "수정 학습"으로 중심 이동**
- CorpGen의 강점은 처음부터 완벽한 정답을 맞추는 능력보다, 실패 신호를 바탕으로 반복 수정하는 능력을 학습한다는 점이다.
- 이는 실제 개발 사이클(테스트 실패 -> 원인 파악 -> 국소 수정)에 직접 대응한다.

2. **데이터 스케일링의 현실성**
- 사람 라벨링만으로는 실제 업무형 데이터를 대규모로 확보하기 어렵다.
- CorpGen처럼 human seed + LLM 확장 파이프라인은 비용 대비 데이터 다양성 확보에 유리하다.

3. **길고 복잡한 컨텍스트 대응**
- 실제 저장소 작업은 여러 파일/의존성/테스트 상태를 동시에 다룬다.
- CorpGen은 이런 조건에서 불필요한 광범위 변경을 줄이고, 국소적이고 목적지향적인 패치를 유도하는 방향으로 해석된다.

4. **평가 지표 재설계 필요성 제기**
- pass/fail 단일 지표는 중간 수준의 실무 가치를 반영하지 못한다.
- 따라서 조직 적용 시에는 "완전 성공률" 외에 "부분 해결 기여도", "리뷰어 수정 부담", "회귀 리스크" 같은 보조 지표가 필요하다.

## 5) 실무 적용 관점 체크리스트
1. 사내/팀 코드베이스 기준의 **실패-수정 로그**(테스트 실패, CI 실패, 코드리뷰 코멘트)를 학습 데이터로 구조화한다.
2. 모델 평가를 pass/fail만 보지 말고 **부분 기여 지표**(수정량, 회귀율, 리뷰 통과까지 추가 수정 횟수)와 함께 본다.
3. PR 생성 자동화 시 "큰 범위 수정"을 억제하는 가드레일(파일 수, diff size, 핵심 파일 보호)을 둔다.
4. 테스트/정적분석/형식검사 결과를 프롬프트 피드백 루프에 자동 주입해 반복 수정 품질을 높인다.

## 6) 한계와 주의사항
1. 블로그 수치는 특정 실험 조건/모델/셋업 기반이므로, 팀 환경에 그대로 재현된다고 가정하면 안 된다.
2. 자동 생성 데이터는 편향/중복/노이즈가 섞일 수 있어 품질 필터링 단계가 필수다.
3. 장문 컨텍스트 능력이 높아져도, 저장소별 규칙(아키텍처/코드스타일/도메인 제약) 미반영 시 실무 적합도는 떨어질 수 있다.

## 7) 결론
CorpGen의 핵심 가치는 "모델이 코드를 처음부터 완성하게 만드는 것"보다 "실패를 빠르게 수정하며 실무 흐름에 맞춰 점진적으로 정답에 수렴하게 만드는 것"에 있다. 따라서 도입 포인트도 단일 점수 경쟁보다, 실제 개발 파이프라인에서 리뷰/수정 비용을 얼마나 줄이는지로 잡는 것이 타당하다.
